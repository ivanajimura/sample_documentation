src/GUI Documentation

This document pertains to the src/GUI folder, which contains modules referring to the user interface of CDG. Our team did not focus on having a great interface, rather, we focused on having a 'good-enough' interface that would not consume numerous hours to develop.

Are you lost... maybe?

This document is part of a series of documents to help onboard new developers to the software Certifiable Documentation Generator (hereby referred to as CDG). This is not a good starting point to understand CDG, so if you have not yet read the following documents, you are highly recommended to do so before continuing reading this one.

Modules

/images Folder

This folder contains the icon in different formats, to be used with different OSes

/completeDPGUI.py

This module prepares the issues to generate a DP of a whole project. This module was developed largely by the 2021-2023 cohort and there are some unnecessary lines of code, as well as some inconsistencies in formatting.

/complianceCheckerGUI.py

This module contains the GUI for the Compliance Checker page.

/logGUI.py

This module handles the logging area to the right of the main page. We knowingly created some technical debt by initializing the logGUI in this module instead of initializing it in the mainGUI and then passing it to other modules. This would require changing the behavior of all the modules in order to receive the logGUI object, so we decided to keep it contained to its own file.

/mainGUI.py

This module is responsible for creating the home page of CDG and for the parent frame in Tkinter that will hold all other pages. This was one of the oldest modules developed by the team, and the first one using Tkinter, so there is likely a number of asinine design choices and a disregard for best practices.

/settingsGUI.py

This module controls the Settings page. The page itself was modeled after the environment.json file, which was used in v1.0. Some of the settings are not actually used by CDG yet, but might be needed for future implementations. A known bug in this module is when a user enters the Settings page, changes one or more fields, does not save, returns to the main page, then returns to the Settings page. The page will show the fields the way the user entered them (this is stored in the textbox variable), but these are not saved in the Storage (so if the user tries to generate a Data Package, the settings from Storage will be used, not the ones displayed in the textboxes). If the user closes the program, they will not receive any notifications of unsaved settings, and the unsaved changes will be lost.

/singleFeatureOrEpicGUI.py

This module is used to create the pages for Single Feature, Single Epic and Single Version. This module has one of the highest Cognitive Complexity ratings of the codebase, according to Sonarqube, but we decided against refactoring it, as it is working mostly as intended. Mostly, because there is a known bug. Let's assume the settings are correct and project XYZ is set. The user enters any of the pages created by this module. CDG will display the epics/features/versions present in project XYZ. If the user backs out to the main menu, goes into Settings, changes the project to ABC, saves the settings, then returns to the same page they had accessed before, the list will show the epics/features/versions from projects XYZ as well as the ones in ABC.
