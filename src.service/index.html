src/service/ Documentation

This document pertains to the src/service folder, which contains modules related to connection to external services, such as Jira and BitBucket. The Compliance Checker module is also inside this folder.

Are you lost... maybe?

This document is part of a series of documents to help onboard new developers to the software Certifiable Documentation Generator (hereby referred to as CDG). This is not a good starting point to understand CDG, so if you have not yet read the following documents, you are highly recommended to do so before continuing reading this one.

Modules

/bitbucket/

bitbucketService.py handles the connection to BitBucket, while bitBucket.py has methods related to getting the data and files.

/jira/

The division of responsibilities among the files is not very clear in this folder, as there are three files: jira.py , jiraRequests.py, jiraService.py. Some modules are deprecated or have never been used. Some requests to Jira API are written in JQL, which might not be familiar to most developers. One known issue is that Jira Server instances do not have a standard field for Epics. It is a custom field, and the id varies from instance to instance (and maybe from project to project). We did not know this when we developed the functionalities related to epics, so we used the id from the mock project we created in the Jira instance offered by UC-DEI. In this case, the field is customfield_10102 . We recommend changing this into a setting and using the stored value in the construction of the JQL query for better compatibility outside the scope of the mock project.

/compliance_checker/

If you are coming across the term 'compliance checker' for the first time in this document, you should read other documents first. Compliance Checker mostly connects to other services, but its placement in this folder is still debateable. The rules folder contains all the rules implemented so far. The majority refer to locating files inside a specific folder in the commit, but other rules have different behavior. The compliance checker class itself is responsible for orchestrating the verification of the rules.
